# SeedRank(OpenSeed) 통합 기능 명세서

## 1. 문서 목적

이 문서는 프로젝트 기획서, 기존 PRD, 포인트 제도, 개발 Task, 수정된 페이지별 기능 명세서와 확정된 병합 결정을 하나의 개발 기준으로 통합한다. 우선순위는 Must, Should, Later로 구분한다.

## 2. MVP 범위

### 2.1 포함

- 이메일 회원가입·로그인·이메일 인증
- 공개 프로필 아이디
- 회사 대표 이메일 기반 기업 인증
- 아이디어 등록·조회·수정·버전 관리
- AI 후보 5개와 기획서 초안 생성
- 대표 콘셉트 이미지 1장 생성과 실패 대체 이미지
- 공개 범위 4종과 권한별 열람
- 검증 질문 1~3개
- 좋아요·조회·구조화 피드백·피드백 채택
- 기여자 보상과 성장 타임라인
- Seed Point 지갑·원장
- Seed Unit 구매·24시간 잠금·회수·회수 가능 잔액
- 랭킹·급상승·최신 정렬과 제목·키워드 검색
- 기업 관심·상세 열람 요청·비동기 1:1 문의
- 신고 접수·사용자 차단

### 2.2 후속 개발

- 유사 아이디어 연결
- 스폰서 광고·결제·정산
- 관리자 백오피스
- 카테고리·공개 범위·성장 단계 필터
- 기업명·복합 태그 검색
- 소셜 로그인·SSO
- 실시간 채팅·푸시 알림
- AI 상세 이미지 다중 생성
- 포인트 만료

## 3. 사용자 역할과 권한

| 역할 | 조회 | 피드백·좋아요 | Seed Unit | 아이디어 | 기업 기능 |
|---|---|---|---|---|---|
| Guest | 공개 콘텐츠 | 불가 | 불가 | 불가 | 불가 |
| User | 권한 범위 내 | 이메일 인증 후 가능 | 이메일 인증 후 가능 | 등록·본인 글 수정 | 불가 |
| Company | User와 동일 | 가능 | 가능 | 등록 가능 | 관심·열람 요청·문의 |
| Operator | DB·내부 도구 | 신고 확인 | 원장 확인 | 운영 확인 | 운영 확인 |

MVP에는 별도의 Admin 화면과 Admin 역할 기반 UI를 구현하지 않는다.

## 4. 페이지 구조

| Route 예시 | 페이지 | 핵심 기능 |
|---|---|---|
| `/` | 메인·랭킹 | 검색, 정렬, 랭킹 카드, 급상승, 기여자 Top 5 |

| `/mypage` | 마이페이지 | 프로필, 지갑, Unit, 피드백, 기여, 회수 잔액 |

### 5.9 Seed Unit 구매·가격·회수

| ID | 요구사항 | 우선순위 |
|---|---|---|
| UNIT-01 | 새 아이디어의 초기 가격은 1 Unit당 10P다. | Must |
| UNIT-02 | 사용자는 정수 Unit 수를 선택하고 현재가 기준 총 결제액을 확인한다. | Must |
| UNIT-03 | 1회 결제액은 최대 100P, 하루 결제액은 최대 300P다. | Must |
| UNIT-04 | 사용자별 아이디어 1개당 활성 보유 원금은 300P를 넘을 수 없다. | Must |
| UNIT-05 | 본인 아이디어의 Seed Unit을 구매할 수 없다. | Must |
| UNIT-06 | 구매 Lot별 구매 시각·Unit·가격·원금을 기록한다. | Must |
| UNIT-07 | 각 Lot은 구매 후 24시간이 지나야 회수 가능하다. | Must |
| UNIT-08 | 회수 금액은 회수 시점의 `Unit × 현재가`로 계산한다. | Must |
| UNIT-09 | 회수 요청 시 가격을 확정하고 Unit을 소각한다. | Must |
| UNIT-10 | 하루 지갑 회수 지급액은 500P를 넘을 수 없다. | Must |
| UNIT-11 | 지갑·일일 한도로 지급하지 못한 실현 금액은 회수 가능 잔액에 저장한다. | Must |
| UNIT-12 | 회수 가능 잔액은 이후 가격 변동의 영향을 받지 않는다. | Must |
| UNIT-13 | 사용자는 조건이 충족되면 대기 잔액의 지갑 지급을 다시 요청할 수 있다. | Must |
| UNIT-14 | 현재가는 하루 1회 마감 랭킹 점수 변화율로 갱신한다. | Must |
| UNIT-15 | 일일 가격 변화율은 -10%~+10%, 절대 가격은 1P~100P로 제한한다. | Must |
| UNIT-16 | 현재가는 랭킹 입력값으로 다시 사용하지 않는다. | Must |
| UNIT-17 | 기존 성과 조건형 예측 보상은 제공하지 않는다. | Must |
| UNIT-18 | 구매·보유·회수 화면에 비금전성 안내를 표시한다. | Must |

### 5.10 랭킹

| ID | 요구사항 | 우선순위 |
|---|---|---|
| RANK-01 | 활성 투자 원금에 로그 함수를 적용해 투자 점수를 계산한다. | Must |
| RANK-02 | 고유 활성 투자자 수를 별도 점수로 반영한다. | Must |
| RANK-03 | 인증 기업 관심 수를 반영한다. | Must |
| RANK-04 | 피드백 수와 채택 피드백 수를 반영한다. | Must |
| RANK-05 | 좋아요와 조회수를 상한과 로그 보정으로 반영한다. | Must |
| RANK-06 | 최근 24시간 투자자·피드백 증가율을 반영한다. | Must |
| RANK-07 | 등록 7일 후 하루 2% 시간 감쇠를 적용한다. | Must |
| RANK-08 | 성장 단계 상승 시 감쇠를 한 차례 완화할 수 있다. | Should |
| RANK-09 | 1~5분 주기로 RankingSnapshot을 생성한다. | Must |
| RANK-10 | 가격 계산용 일 마감 스냅샷을 별도로 보존한다. | Must |

초기 점수식은 다음과 같다.

```text
investmentScore = log(activePrincipal + 1) × 35
diversityScore = min(uniqueActiveInvestors × 3, 20)
companyScore = min(companyInterestCount × 12, 36)
feedbackScore = min(feedbackCount × 2 + acceptedFeedbackCount × 5, 30)
reactionScore = min(likeCount × 0.8 + log(viewCount + 1) × 2, 20)
growthScore = min(normalized24hGrowth, 15)
totalScore = investmentScore + diversityScore + companyScore
           + feedbackScore + reactionScore + growthScore - decayScore
```

## 7. 핵심 계산과 처리 순서

### 7.1 Unit 구매

1. 인증·권한·자기 아이디어 여부를 검사한다.
2. 현재가와 구매 Unit 수로 결제액을 계산한다.
3. 1회·일일·아이디어별 원금 한도와 지갑 잔액을 검사한다.
4. PointLedger에 구매 차감을 기록한다.
5. SeedUnitLot을 생성하고 `unlockedAt = purchasedAt + 24시간`으로 저장한다.
6. 랭킹 재계산 이벤트를 발생시킨다.

### 7.2 Unit 회수

1. 회수 가능한 Lot과 Unit 수를 확인한다.
2. 현재가로 실현 포인트를 계산하고 해당 Unit을 소각한다.
3. 오늘 지급 가능한 한도와 지갑 여유 공간을 계산한다.
4. 즉시 지급분은 PointWallet과 PointLedger에 반영한다.
5. 나머지는 pendingRecoveryBalance와 회수 원장에 기록한다.
6. 실현 시점의 가격을 고정하고 이후 가격 변동을 적용하지 않는다.

### 7.3 회수 가능 잔액 지급

1. pendingRecoveryBalance가 0보다 큰지 확인한다.
2. 일일 회수 잔여 한도와 지갑 여유 공간 중 작은 금액을 지급한다.
3. 지급액만큼 pendingRecoveryBalance를 차감하고 원장을 기록한다.

### 7.4 일일 가격 갱신

1. 전일과 당일의 마감 RankingSnapshot을 조회한다.
2. 원시 점수 변화율을 계산한다.
3. 변화율을 -10%~+10%로 제한한다.
4. 이전 가격에 적용하고 정수로 반올림한다.
5. 가격을 1P~100P 범위로 제한한다.
6. UnitPriceSnapshot과 Idea.currentUnitPrice를 갱신한다.

## 9. 비기능 요구사항

- 포인트 차감·회수·대기 잔액 처리는 DB 트랜잭션으로 원자성을 보장한다.
- PointLedger는 append-only로 관리하고 기존 원장 행을 수정·삭제하지 않는다.
- 가격 갱신과 랭킹 배치는 동일 아이디어에 중복 실행되지 않도록 잠금 또는 idempotency key를 사용한다.
- 비밀번호, 인증 토큰과 기업 이메일은 로그에 평문으로 남기지 않는다.
- 공개 범위는 프론트엔드 숨김이 아니라 API 권한 검사로 강제한다.
- 모든 폼은 키보드 접근, label, 오류 메시지와 모바일 조작을 지원한다.
- Seed Unit 관련 화면은 현금화 불가와 비금전성 고지를 항상 표시한다.

## 10. QA 수용 기준

- 이메일 미인증 사용자의 피드백·Unit 구매가 차단된다.
- 본인 아이디어 Unit 구매가 차단된다.
- 구매 후 24시간 이전 회수가 차단되고 이후 정상 회수된다.
- 회수 시점 가격으로 손익이 계산되고 가격이 이후 바뀌어도 실현 금액은 변하지 않는다.
- 지갑·일일 회수 한도 초과분이 회수 가능 잔액으로 정확히 이동한다.
- PointLedger 합계와 PointWallet·대기 잔액이 일치한다.
- 가격 변화가 일일 ±10%, 절대 1P~100P 범위를 넘지 않는다.
- 현재가 변경이 랭킹 입력값으로 재사용되지 않는다.
- 공개 범위별 API 응답이 권한 규칙과 일치한다.
- 기여자가 원본 아이디어를 직접 수정할 수 없다.
- 채택 피드백 기반 새 버전과 타임라인이 생성된다.
- 기업 상세 열람 승인과 비동기 메시지 읽음 상태가 정상 동작한다.
- 차단한 사용자의 피드백과 메시지가 숨겨진다.
