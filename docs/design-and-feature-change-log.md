# OpenSeed 단계별 디자인·기능 변경 상세 문서

> 작성일: 2026-07-16  
> 대상 프로젝트: `openseed-fe`  
> 기준: 현재 작업 트리의 구현 상태  
> 참고: 6단계 기업 문의 페이지에는 확인된 레이아웃 미해결 사항이 남아 있다.

## 1. 문서 목적

이 문서는 OpenSeed 프론트엔드에서 단계별로 수행한 디자인 개선과, 디자인 변경 이후 실제로 동작하도록 연결된 기능을 정리한다. 단순히 화면이 어떻게 달라졌는지만 기록하지 않고 다음 항목을 함께 다룬다.

- 각 단계의 개선 목표
- 화면 구조와 시각적 위계 변화
- 사용자 조작에 따른 실제 동작
- React Query, 라우팅, mock API와의 연결
- 반응형 처리
- 현재 mock 단계의 제약
- 테스트 및 미해결 사항

## 2. 전체 변경 요약

| 단계 | 대상 | 핵심 결과 | 상태 |
|---|---|---|---|
| 1 | 홈·랭킹 | 섹션 경계를 줄이고 전체 랭킹 안에서 상위 3개만 차등 강조 | 완료 |
| 2 | 아이디어 상세 | 활성 원금을 대표 수치로 만들고 본문·검증·기업 참여 구조 재편 | 완료 |
| 3 | 헤더·내비게이션 | 발견 중심 내비게이션, 검색, 인증 상태별 액션, 프로필 메뉴 구성 | 완료 |
| 4 | AI 아이디어 생성 | 6단계 생성 흐름과 AI 보조 패널을 하나의 편집 작업 공간으로 구성 | 완료 |
| 5 | 마이페이지 | 활동 중심 정보 구조, 지갑·보유 Unit·기업 활동·프로필 편집 구현 | 완료 |
| 6 | 기업 문의 | 아이디어 문맥과 문의 상태를 포함한 메시지 기능 구현 | 기능 완료, 레이아웃 이탈 미해결 |
| 7 | 인증 화면 | 로그인·가입·이메일 인증·기업 인증 디자인 통일 | 완료 |

공통적으로 흰색 카드의 과도한 반복을 줄이고, 밝은 회청색 계열 배경과 낮은 대비의 선, 제한적인 그림자, 파란색 핵심 액션을 사용했다. 전체 배경의 그라데이션은 모션 없이 고정된 상태이며, 상위 랭킹의 금·은·동 그림자와 충돌하지 않도록 채도를 낮췄다.

## 3. 1단계 — 홈·아이디어 랭킹

### 3.1 개선 목표

- 헤더와 히어로, 현황 수치, 랭킹이 각각 별도 카드처럼 끊겨 보이던 문제를 완화한다.
- 사용자가 홈 진입 직후 아이디어 탐색과 랭킹을 한눈에 인지하게 한다.
- 상위 3개를 별도 랭킹으로 분리하지 않고 전체 랭킹 흐름 안에서만 강조한다.
- 숫자를 지나치게 키우거나 굵게 만들지 않고 색상, 정렬, 여백으로 가시성을 높인다.

### 3.2 디자인 변경 사항

#### 히어로 영역

- 기존 히어로 카드의 테두리, 둥근 박스, 강한 그림자를 제거했다.
- 히어로 카피와 서비스 현황 수치를 하나의 넓은 레이아웃 안에 배치했다.
- 핵심 문장, 설명, `AI로 아이디어 만들기`, `랭킹 둘러보기` 버튼의 우선순위를 유지했다.
- 현황 수치는 세로 3행으로 정렬했다.
  - 검증 중인 아이디어
  - 활성 Seed Point
  - 기업 관심 신호
- 항목명과 숫자를 좌우 두 열로 정렬해 숫자 열이 일정한 축을 갖도록 했다.
- 숫자에는 OpenSeed의 파란색을 사용하고 `font-variant-numeric: tabular-nums`를 적용해 자리수가 달라도 안정적으로 보이게 했다.
- 수치를 감싸는 별도 배경 카드, 아이콘, 모션은 사용하지 않았다.

#### 전체 배경

- 전체 페이지에 은은한 회청색 그라데이션을 적용했다.
- 실험했던 움직이는 블루 방사형 그라데이션은 제거했다.
- 현재 배경은 고정되어 있으며 콘텐츠와 상위 랭킹 그림자를 방해하지 않는다.

#### 랭킹 영역

- 상위 3개와 나머지 아이디어를 별도 섹션으로 나누지 않고 `전체 랭킹` 한 목록에서 보여준다.
- 모든 아이디어는 동일한 정보 구조를 사용한다.
  - 순위
  - 카테고리
  - 성숙도
  - 제목
  - 요약
  - 참여자 수
  - 기업 관심 수
  - Seed Unit 현재가
  - 24시간 추세
- 1·2·3위의 테두리 색과 순위 숫자 색은 일반 항목과 동일하게 유지했다.
- 상위 3개에만 차등 그림자를 적용했다.
  - 1위: 금색 계열 그림자
  - 2위: 은색 계열 그림자
  - 3위: 동색 계열 그림자
- 상위 3개의 순위 숫자 위에 왕관 아이콘을 추가했다. 왕관만 금·은·동 계열 색상을 사용한다.
- 그림자가 인접 행과 겹쳐 보이지 않도록 상위 항목의 상하 간격을 별도로 확보했다.
- 카테고리 필터는 가로 스크롤 가능한 칩 형태로 구성했다.
- 랭킹 헤더에는 현재 필터 결과의 아이디어 개수를 표시한다.

### 3.3 기능 동작

- 헤더 검색 또는 홈 검색 조건은 URL의 `q` 파라미터에 반영된다.
- 정렬 선택은 `sort` 파라미터로 관리된다.
  - 랭킹순
  - 급상승순
  - 최신순
- 카테고리 선택은 `category` 파라미터로 관리된다.
- 필터 변경 시 URL 상태가 갱신되므로 브라우저 이동과 새로고침 후에도 조건을 표현할 수 있다.
- 랭킹 행 전체가 아이디어 상세 페이지 링크로 동작한다.
- 결과가 없으면 빈 상태 안내를 표시한다.
- 히어로의 수치는 조회한 아이디어 데이터로 계산된다.
  - 아이디어 수: 배열 길이
  - 활성 Seed Point: 모든 아이디어의 `seedInvested` 합계
  - 기업 관심 신호: 모든 아이디어의 관심 기업 수 합계

### 3.4 반응형 처리

- 태블릿 이하에서는 히어로 카피와 수치 영역을 세로 배치한다.
- 모바일에서는 제목 크기와 여백을 줄이고, 수치 레이아웃의 열 너비를 축소한다.
- 카테고리 필터는 줄바꿈 대신 가로 스크롤한다.
- 랭킹 요약은 모바일에서 두 줄 말줄임 처리한다.
- 추세 정보는 랭킹 본문 아래 행으로 이동한다.

### 3.5 관련 파일

- `src/App.tsx`
- `src/styles.css`

## 4. 2단계 — 아이디어 상세 페이지

### 4.1 개선 목표

- 제목의 과도한 굵기와 밀도를 낮춰 장문 열람 피로도를 줄인다.
- 가장 중요한 참여 지표를 `활성 원금`으로 명확히 한다.
- 기존 활성 원금 위치에 있던 보조 수치를 `Seed Unit 현재가`로 변경한다.
- 문제 정의 → 목표 고객 → 해결 방법 → 수익 모델을 번호 중심의 연속 본문이 아니라 내용 중심 구조로 보여준다.
- 검증 질문, 기업 관심, 제안자 문의를 각각 독립된 기능으로 구분한다.

### 4.2 디자인 변경 사항

#### 상세 상단

- 카테고리, 공개 범위, 성숙도를 상단 배지로 정리했다.
- 제목의 굵기와 자간을 낮추고 줄 간격을 넓혔다.
- 요약, 작성자, 좋아요, 신고 기능을 제목 아래 메타 정보로 묶었다.
- 오른쪽에는 sticky 참여 패널을 배치했다.

#### 대표 수치와 참여 패널

- 대표 수치를 `활성 원금`으로 표시한다.
- 활성 원금 아래에 추세 값을 함께 보여준다.
- 보조 수치는 다음 세 항목으로 정리했다.
  - Seed Unit 현재가
  - 참여자 수
  - 기업 관심 수
- 참여 가능한 아이디어에는 `Seed Unit 참여하기` 버튼을 표시한다.
- 권한 또는 아이디어 상태상 참여할 수 없으면 비활성 안내 문구를 표시한다.
- 가상 포인트가 금전적 권리나 수익 배분을 의미하지 않는다는 안내를 유지한다.

#### 아이디어 본문

- `IDEA STORY` 도입부를 추가했다.
- 각 항목은 번호 박스 없이 제목과 본문 중심의 독립된 읽기 블록으로 구성했다.
- 항목 사이에는 충분한 세로 여백과 얇은 구분선만 사용했다.
- 본문은 줄 간격을 넓혀 긴 설명을 쉽게 읽을 수 있게 했다.
- 공개 권한이 `FULL`이 아닐 경우 해결 방법과 수익 모델은 숨기고 공개 가능한 범위만 표시한다.

#### 검증 질문·기업 관심·1:1 문의

오른쪽 validation stack 안에서 세 기능을 각각 별도 카드로 분리했다.

1. `함께 검증할 질문`
   - 아이디어별 검증 질문을 목록으로 표시한다.
   - `피드백으로 답하기`를 누르면 피드백 탭으로 이동한다.
2. `기업 관심`
   - 관심 기업 수를 표시한다.
   - 기업 이름의 첫 글자를 겹친 프로필 형태로 표시한다.
   - 권한이 있는 기업은 `기업 관심 추가하기`를 실행할 수 있다.
3. `제안자 1:1 문의`
   - 제안자의 프로필과 역할을 표시한다.
   - `1:1 문의하기`를 누르면 `/messages?idea={ideaId}`로 이동한다.

### 4.3 기능 동작

#### 탭

- 아이디어 소개
- 피드백
- 성장 기록
- 기업 관심

탭 상태는 페이지 내부 상태로 전환되며 각 탭에 필요한 데이터 표현을 유지한다.

#### 좋아요와 신고

- 좋아요 버튼은 API 요청 후 아이디어 상세 query를 갱신한다.
- 신고 버튼은 mock 신고 API를 호출한다.
- 신고 완료 시 콘텐츠가 자동 삭제되지 않는다는 안내를 표시한다.

#### Seed Unit 참여

- 참여 버튼을 누르면 구매 확인 모달이 열린다.
- Unit 수량에 따라 필요 포인트를 계산한다.
- 구매 성공 후 다음 query를 갱신한다.
  - 사용자 포인트
  - 보유 Seed Unit
- 구매 권한은 `idea.capabilities.purchase`에 따라 제어된다.

#### 공개 범위와 열람 요청

- `contentAccess`가 `FULL`이 아니면 잠금 안내를 표시한다.
- 매칭형 공개 등 열람 요청이 가능한 경우 작성자에게 메시지를 포함한 상세 열람 요청을 보낼 수 있다.
- 이미 대기 중인 요청이 있으면 mock API가 중복 요청 오류를 반환한다.

#### 구조화 피드백

- 피드백 유형을 선택할 수 있다.
- 본문은 100자 이상이어야 등록 버튼이 활성화된다.
- 선택적으로 근거 URL 또는 설명을 입력할 수 있다.
- 등록 성공 후 입력값을 초기화하고 상세 데이터와 사용자 데이터를 다시 조회한다.
- 삭제되지 않은 피드백만 표시한다.
- 채택된 피드백을 우선 정렬한다.
- 본인 피드백은 수정·삭제할 수 있다.
- 아이디어 작성자는 채택 이유와 수정 방향을 10자 이상 입력한 후 피드백을 채택할 수 있다.
- 피드백 채택 시 mock 데이터에서 성장 기록과 기여 보상이 갱신된다.

### 4.4 반응형 처리

- 태블릿 이하에서 상세 상단과 참여 패널이 한 열로 전환된다.
- sticky 패널은 작은 화면에서 일반 흐름으로 변경된다.
- 본문과 검증 stack도 한 열로 변경된다.
- 모바일에서 제목, 본문, 탭, 패널의 크기와 여백을 축소한다.

## 5. 3단계 — 헤더·내비게이션·세션 상태

### 5.1 개선 목표

- 데스크톱 내비게이션을 아이디어 발견 중심으로 단순화한다.
- 와디즈 계열 레퍼런스처럼 헤더의 경계와 무게를 최소화한다.
- 로그인 여부에 따라 헤더 액션을 자연스럽게 전환한다.
- 로그인 사용자가 프로필 메뉴에서 주요 기능과 로그아웃을 처리할 수 있게 한다.

### 5.2 디자인 변경 사항

- 헤더 높이를 68px로 줄였다.
- 배경은 반투명 흰색과 blur를 사용하고 하단 경계선은 매우 낮은 대비로 설정했다.
- 로고 마크 크기와 내비게이션 글자 크기를 축소했다.
- 데스크톱의 대표 내비게이션은 `아이디어 발견`에 집중한다.
- 헤더 검색창은 회색 배경의 미니멀한 입력창으로 구성했다.
- `아이디어 등록`은 파란색 핵심 CTA로 분리했다.
- 세션 조회 중에는 shimmer 형태의 로딩 자리를 표시한다.

### 5.3 비로그인 상태 동작

- 프로필 영역 대신 다음 링크를 표시한다.
  - 로그인 → `/login`
  - 회원가입 → `/signup`
- 인증이 필요한 페이지 접근 시 route guard가 로그인 페이지로 이동시킨다.
- `RequireAuth`는 원래 접근하려던 경로를 location state의 `from`으로 보존한다.
- 로그인 성공 후 보존된 경로가 있으면 해당 경로로 돌아간다.

### 5.4 로그인 상태 동작

- 보유 포인트를 헤더에 표시한다.

## 6. 4단계 — AI 아이디어 생성 흐름

### 6.1 개선 목표

- 단계별 카드가 분절되어 보이던 생성 흐름을 하나의 편집 작업 공간으로 통합한다.
- 사용자가 현재 단계, 다음 행동, 기획서 완성도를 항상 알 수 있게 한다.
- AI 생성 결과를 그대로 게시하는 흐름이 아니라 사용자가 선택하고 편집하는 흐름을 강조한다.

### 6.2 디자인 변경 사항

#### 생성 페이지 헤더

- `AI IDEA BUILDER` 레이블과 페이지 목적을 표시한다.
- `초안 자동 저장됨` 상태를 명시한다.
- 기존의 큰 원형 stepper 대신 `현재 단계 / 전체 단계`, 진행 막대, 단계명을 표시하는 간결한 progress bar를 사용한다.

#### 작업 공간

- 왼쪽은 단계별 편집 surface, 오른쪽은 sticky `AI CO-PILOT` 패널로 구성했다.
- 입력 surface는 동일한 여백, 경계, 포커스 스타일을 사용한다.
- AI CO-PILOT은 단계별 도움말, 완성도, 현재 입력 미리보기를 표시한다.
- 모바일에서는 한 열로 전환하고 AI 안내를 편집 영역 위에 배치한다.

### 6.3 6단계 기능 흐름

#### 1단계 — 문제 입력

- 사업화 키워드와 문제 상황을 입력한다.
- 키워드가 비어 있거나 문제 상황이 10자 미만이면 AI 생성 버튼이 비활성화된다.
- 제출하면 AI 생성 job을 생성하고 2단계로 이동한다.

#### 2단계 — AI 생성

- job 상태를 700ms 간격으로 polling한다.
- 상태가 `COMPLETED`가 되면 polling을 중단한다.
- 완료 후 5개 후보 확인 버튼이 활성화된다.
- 실패하면 다시 시도할 수 있다.

#### 3단계 — 후보 선택

- 5개 후보를 서로 다른 사업화 관점으로 표시한다.
  - 시장성 중심
  - 실행 가능성 중심
  - 커뮤니티 중심
  - 확장성 중심
  - 차별화 중심
- 후보를 선택하면 해당 후보의 제목, 요약, 문제, 고객, 해결 방법, 수익 모델을 편집 form으로 복사하고 4단계로 이동한다.

#### 4단계 — 기획서 편집

- 다음 내용을 직접 수정한다.
  - 아이디어 제목
  - 한 줄 요약
  - 문제 정의
  - 목표 고객
  - 해결 방법
  - 수익 모델
- 6개 핵심 필드 중 채워진 수를 기준으로 기획서 완성도를 계산한다.

#### 5단계 — 공개 범위·검증 질문

- 공개 범위를 선택한다.
  - 공개형
  - 반공개형
  - 비공개 검증형
  - 매칭형 공개
- 각 공개 범위의 의미를 선택 옵션 안에서 설명한다.
- 검증 질문은 줄바꿈 단위로 입력하며 최대 3개까지만 저장한다.

#### 6단계 — 최종 미리보기·게시

- 공개 범위, 제목, 요약, 문제, 고객, 해결 방법, 수익 모델을 게시 형태로 미리 본다.
- 내용 수정으로 돌아가거나 게시할 수 있다.
- 게시 성공 시 draft를 제거하고 생성된 아이디어 상세 페이지로 이동한다.

### 6.4 초안 저장

- 편집 form은 `sessionStorage`의 `idea-draft` 키에 자동 저장된다.
- 같은 브라우저 탭 세션에서 페이지를 다시 열면 저장된 초안을 불러온다.
- 게시 성공 시 저장된 초안을 삭제한다.

### 6.5 현재 AI mock 동작

- 실제 OpenAI, Anthropic, Google 등의 외부 LLM은 연결되어 있지 않다.
- MSW의 `/api/ai/idea-jobs` handler가 mock AI 역할을 한다.
- 키워드와 문제 상황을 조합해 규칙 기반으로 5개 후보를 만든다.
- 첫 polling에서는 `PROCESSING`, 이후에는 `COMPLETED` 상태를 반환한다.
- 키워드에 `실패`가 포함되면 실패 상태를 재현할 수 있다.
- 따라서 mock 단계에서도 비동기 생성, polling, 성공, 실패, 재시도 UI를 확인할 수 있다.

## 7. 5단계 — 마이페이지·프로필

### 7.1 개선 목표

- 단순 자산 목록보다 사용자의 검증 활동과 기여를 먼저 보여준다.
- 포인트, 보유 Seed Unit, 피드백, 기업 활동을 명확한 탭으로 구분한다.
- 프로필 정보를 실제로 수정하고 즉시 화면에 반영할 수 있게 한다.
- 프로필 이미지가 없을 때도 일관된 기본 프로필을 제공한다.

### 7.2 디자인 변경 사항

#### 프로필 상단

- 프로필 이미지, 이름, 공개 프로필 ID를 왼쪽에 배치했다.
- 이메일 인증과 기업 인증 상태를 배지로 표시한다.
- `프로필 수정` 버튼을 상단 액션으로 제공한다.

#### 활동 요약

- `나의 검증 활동` 메시지를 중심으로 다음 수치를 표시한다.
  - 작성한 피드백
  - 채택된 기여
  - 참여 아이디어
- 포인트 지갑은 같은 영역 안의 독립된 요약 블록으로 제공한다.
  - 사용 가능 포인트
  - 회수 가능 잔액

#### 탭 구조

- 내 활동
- 포인트·내역
- Seed Unit
- 기업 활동 — 기업 계정에만 표시

### 7.3 내 활동 기능

- 전체 아이디어에서 현재 사용자 이름과 일치하는 피드백을 모아 표시한다.
- 각 피드백은 관련 아이디어 상세 페이지로 연결된다.
- 피드백의 상태를 구분한다.
  - 검토 중
  - 채택된 기여
- 채택된 피드백 수와 받은 기여 포인트를 요약한다.
- 현재 mock에서는 채택 1회당 100P 기준으로 표시한다.

### 7.4 포인트·Seed Unit 기능

- 포인트 원장은 활동명, 변동 포인트, 일시를 표시한다.
- 보유 Seed Unit 카드에는 다음 정보를 표시한다.
  - 잠금 또는 회수 가능 상태
  - 구매 시각
  - 보유 Unit
  - 매수 원금
  - 현재 가치
  - 손익
- 24시간 잠금 상태인 보유분은 회수 버튼이 비활성화된다.
- 회수 가능한 보유분은 먼저 회수 견적을 조회한다.
- 견적 모달에서 다음 내용을 확인하고 회수를 확정한다.
  - 회수 Unit
  - 확정 현재가
  - 실현 포인트
  - 즉시 지갑 지급액
  - 회수 가능 잔액
- 회수 후 보유 내역, 사용자 포인트, 포인트 원장을 갱신한다.
- 회수 가능 잔액이 있으면 지갑으로 받는 버튼을 표시한다.

### 7.5 기업 활동 기능

- 로그인 계정의 role이 `COMPANY`일 때만 탭을 표시한다.
- 현재 기업이 관심을 표시한 아이디어 목록을 보여준다.
- 항목을 선택하면 아이디어 상세 페이지로 이동한다.

### 7.6 프로필 수정 기능

- `프로필 수정`을 누르면 현재 정보를 불러온 모달이 열린다.
- 다음 정보를 수정할 수 있다.
  - 이름
  - 공개 프로필 ID
  - 프로필 이미지
- 입력 즉시 모달의 프로필 미리보기에 반영된다.
- 이름은 2자 이상이어야 한다.
- 프로필 ID는 영문, 숫자, 밑줄 조합 3~20자여야 한다.
- 이미지 파일만 선택할 수 있다.
- 이미지 크기는 2MB 이하로 제한한다.
- 이미지는 FileReader로 Data URL로 변환해 mock 프로필에 저장한다.
- `기본 프로필로 변경`을 누르면 업로드 이미지를 제거한다.
- 저장 성공 시 React Query의 사용자 캐시를 갱신하고 모달을 닫는다.
- 헤더와 마이페이지의 프로필이 즉시 변경된 정보로 다시 렌더링된다.

### 7.7 기본 프로필 정책

- 프로필 이미지가 없으면 공개 프로필 ID의 첫 글자를 대문자로 표시한다.
- 회원가입 시 다음 팔레트 중 하나를 무작위로 선택한다.
  - 파랑
  - 초록
  - 보라
  - 주황
  - 분홍
  - 청록
- 선택된 `avatarColor`를 사용자 데이터에 저장하므로 같은 mock 세션 동안 색상이 유지된다.

### 7.8 프로필 ID와 내부 고유 ID 정책

- 현재 mock 단계에서는 공개 프로필 ID 중복을 허용한다.
- `profileAvailable()`은 항상 사용 가능 상태를 반환한다.
- 공개 프로필 ID가 같아도 내부 사용자는 별도 `id`로 구분된다.
- 신규 사용자의 내부 ID 형식은 `u-${crypto.randomUUID()}`이다.
- 이메일은 계속 고유해야 하며 중복 가입을 허용하지 않는다.
- 실제 서버 연동 단계에서는 공개 프로필 ID 중복 허용 여부와 검색·멘션 정책을 별도로 확정해야 한다.

## 8. 6단계 — 기업 문의

### 8.1 개선 목표

- 단순 채팅 목록을 아이디어 기반 기업 문의 도구로 확장한다.
- 문의가 어떤 아이디어와 연결되어 있는지 대화 중에도 유지한다.
- 문의 상태와 최근 메시지를 목록에서 빠르게 파악한다.
- 상세 페이지의 `1:1 문의하기`에서 신규 문의를 시작할 수 있게 한다.

### 8.2 디자인 변경 사항

#### 문의 목록

- 기업명, 관련 아이디어 제목, 최근 메시지를 표시한다.
- 읽지 않은 메시지 수를 표시한다.
- 마지막 메시지와 unread 상태를 기준으로 상태 배지를 계산한다.
  - unread가 1개 이상: `답변 필요`
  - unread가 없고 마지막 메시지가 내가 보낸 메시지: `답변 대기`
  - 그 외: `확인 완료`
- 선택된 문의는 흰색 배경과 낮은 그림자로 구분한다.

#### 대화 영역

- 상단에 기업명, 문의 제목, 인증 기업 배지를 표시한다.
- 관련 아이디어 context 영역에 다음 내용을 고정한다.
  - 카테고리
  - 아이디어 제목
  - 요약
  - 상세 페이지 링크
- 대화 메시지는 보낸 사람에 따라 좌우를 구분한다.
- 빠른 답장 문구를 제공한다.
  - 검토 자료를 공유드리겠습니다.
  - 미팅 가능한 일정을 알려주세요.
  - 상세 내용을 확인했습니다.
- 빠른 답장을 선택하면 입력창에 문구가 채워진다.

### 8.3 신규 문의 동작

- 상세 페이지에서 `/messages?idea={ideaId}`로 진입한다.
- 해당 아이디어의 기존 thread가 있으면 그 thread를 자동 선택한다.
- 기존 thread가 없으면 신규 문의 안내 화면을 표시한다.
- 첫 메시지를 전송하면 `/ideas/{ideaId}/inquiries` mock API를 호출한다.
- mock API는 다음 구조의 thread를 생성한다.
  - 고유 thread ID
  - 관련 idea ID
  - mock 기업명 `OpenSeed Labs`
  - 아이디어 제목 기반 문의 제목
  - 첫 메시지
- 생성 성공 후 thread 목록 query를 갱신하고 생성된 대화를 선택한다.
- 동일 아이디어에 이미 thread가 있으면 중복 생성 오류를 반환한다.

### 8.4 기존 문의 동작

- 메시지 입력 후 전송하면 해당 thread의 messages에 새 메시지를 추가한다.
- 전송 성공 후 입력값을 비우고 thread 목록을 다시 조회한다.
- 선택한 thread에 unread가 있으면 읽음 API를 호출한다.
- thread 목록은 45초마다 자동 갱신된다.

### 8.5 확인된 미해결 사항

기업 문의 페이지에는 현재 영역 이탈 문제가 남아 있다.

- 증상: 일부 viewport에서 채팅 하단의 빠른 답장과 입력 영역이 message shell 내부에 안정적으로 유지되지 않거나 화면 아래로 밀려 보인다.
- 시도한 대응:
  - 상태 배지에 적용되던 기존 `aside em` 스타일 충돌 완화
  - flex 자식의 `min-height: 0` 적용
  - message shell 높이를 viewport 기반으로 계산
  - 대화 영역을 5행 grid로 재정의
  - 기존 `.message-shell > section` 선택자보다 구체적인 override 추가

## 9. 7단계 — 로그인·회원가입·인증 화면

### 9.1 개선 목표

- 로그인, 회원가입, 이메일 인증, 기업 인증 화면의 디자인을 통일한다.
- 단순한 중앙 카드보다 서비스 가치와 실제 입력 작업의 영역을 구분한다.
- 입력, 오류, 성공, 진행 상태를 일관되게 표현한다.

### 9.2 공통 AuthShell 변경

- 데스크톱에서 2열 구조를 사용한다.
  - 왼쪽: 브랜드와 OpenSeed 가치 설명
  - 오른쪽: 현재 인증 작업
- 왼쪽에는 다음 내용을 표시한다.
  - AI를 활용한 아이디어 구체화
  - 커뮤니티 기반 가능성 검증
  - 인증 기업과의 안전한 연결
  - 인증 기반 참여 환경 안내
- 오른쪽에는 화면별 kicker, 제목, 설명, form 또는 인증 상태를 표시한다.
- 입력창의 포커스, placeholder, 오류 메시지, 버튼 크기를 통일했다.
- 모바일에서는 1열로 전환하고 왼쪽 설명을 축소한다.

### 9.3 로그인

- 이메일과 비밀번호를 입력한다.
- Zod schema와 React Hook Form으로 검증한다.
- 로그인 중에는 버튼을 비활성화하고 진행 문구를 표시한다.
- 인증 실패 시 API 오류 메시지를 form 안에 표시한다.
- 성공 시 사용자 데이터를 React Query 캐시에 저장한다.
- 보호된 페이지에서 이동한 경우 원래 경로로 돌아간다.
- mock 기본 계정은 다음 값으로 form에 미리 입력되어 있다.
  - 이메일: `minseo@openseed.co.kr`
  - 비밀번호: `password123`

### 9.4 회원가입

- 이름과 공개 프로필 ID를 한 행에 배치한다.
- 비밀번호와 확인 입력을 한 행에 배치한다.
- 모바일에서는 모든 입력이 한 열로 전환된다.
- 입력 검증 규칙:
  - 올바른 이메일 형식
  - 비밀번호 8자 이상
  - 비밀번호 확인 일치
  - 프로필 ID 영문·숫자·밑줄 3~20자
  - 이름 2~20자
- 제출 전 이메일과 프로필 ID 사용 가능 여부를 확인한다.
- 현재 mock에서는 프로필 ID 중복을 허용하므로 항상 사용 가능하다.
- 이메일 중복은 계속 차단한다.
- 가입 성공 시 새 사용자가 현재 session이 되고 이메일 인증 화면으로 이동한다.
- 신규 사용자는 무작위 기본 프로필 색상과 UUID 기반 내부 ID를 받는다.

### 9.5 이메일 인증

- 인증 메일 확인 안내와 재전송 버튼을 제공한다.
- mock token은 `valid-email-token`이다.
- URL에 token이 있으면 해당 링크로 인증 완료 버튼을 표시한다.
- 인증 성공 시:
  - `emailVerifiedAt` 기록
  - 가입 보너스 300P 지급
  - 사용자 query 캐시 갱신
  - 홈으로 이동하는 시작 버튼 표시
- 이미 인증된 사용자에게 중복 보너스가 지급되지 않는다.

### 9.6 기업 인증

- 이메일 인증을 완료한 로그인 사용자만 접근할 수 있다.
- 회사 대표 이메일을 입력해 인증 링크를 요청한다.
- Gmail, Naver, Daum, Kakao, Outlook 등 무료 개인 메일 도메인은 mock 단계에서 거부한다.
- mock token은 `valid-company-token`이다.
- token이 있는 경우 인증 완료 버튼을 표시한다.
- 인증 성공 시:
  - role을 `COMPANY`로 변경
  - `companyVerifiedAt` 기록
  - 사용자 query 캐시 갱신
  - 기업 관심과 1:1 문의 기능을 사용할 수 있다는 완료 안내 표시

### 9.7 Route Guard

- `/mypage`, `/messages`는 로그인이 필요하다.
- `/ideas/new`, `/company/verify`는 로그인과 이메일 인증이 필요하다.
- 세션 조회 중에는 세션 확인 상태를 표시한다.
- 미로그인 사용자는 로그인 페이지로 이동한다.
- 이메일 미인증 사용자는 이메일 인증 페이지로 이동한다.

## 10. 주요 사용자 흐름

### 10.1 비로그인 사용자의 아이디어 탐색

1. 홈에서 아이디어 랭킹과 현황 수치를 확인한다.
2. 검색, 정렬, 카테고리 필터로 아이디어를 좁힌다.
3. 랭킹 행을 선택해 상세 내용을 확인한다.
4. 인증이 필요한 기능에 접근하면 로그인 페이지로 이동한다.
5. 로그인 성공 후 가능한 경우 원래 접근 경로로 돌아간다.

### 10.2 신규 사용자 가입

1. 헤더에서 회원가입을 선택한다.
2. 이름, 프로필 ID, 이메일, 비밀번호를 입력한다.
3. 입력 schema와 이메일 중복 여부를 확인한다.
4. 내부 UUID와 기본 프로필 색상을 가진 mock 사용자가 생성된다.
5. 이메일 인증 화면으로 이동한다.
6. 유효한 token으로 인증하면 300P를 받고 기능이 활성화된다.

### 10.3 AI 아이디어 생성·게시

1. 이메일 인증 사용자로 `/ideas/new`에 접근한다.
2. 키워드와 문제 상황을 입력한다.
3. mock AI job이 생성되고 polling이 시작된다.
4. 생성된 후보 5개 중 하나를 선택한다.
5. 상세 기획서를 수정한다.
6. 공개 범위와 검증 질문을 설정한다.
7. 미리보기 후 게시한다.
8. 생성된 아이디어 상세 페이지로 이동한다.

### 10.4 아이디어 검증 참여

1. 상세 페이지에서 검증 질문을 확인한다.
2. 피드백 탭으로 이동한다.
3. 유형, 100자 이상의 의견, 선택적 근거를 입력한다.
4. 등록하면 상세 데이터와 사용자 활동이 갱신된다.
5. 작성자가 피드백을 채택하면 기여 기록과 보상이 반영된다.
6. 마이페이지에서 작성한 피드백과 채택 여부를 확인한다.

### 10.5 Seed Unit 참여·회수

1. 상세 페이지에서 Unit 수량을 선택해 참여한다.
2. 사용자 포인트가 차감되고 holding이 생성된다.
3. 마이페이지 Seed Unit 탭에서 보유 현황과 손익을 확인한다.
4. 잠금이 해제된 holding의 회수 견적을 조회한다.
5. 회수 모달에서 지급 구조를 확인하고 확정한다.
6. 지갑 한도를 초과하는 금액은 회수 가능 잔액으로 남는다.
7. 회수 가능 잔액 받기를 통해 가능한 범위만 지갑으로 지급한다.

### 10.6 기업 관심·문의

1. 인증 기업 사용자가 아이디어 상세 페이지에서 관심을 추가한다.
2. 관심 기업 수와 기업 목록이 갱신된다.
3. `1:1 문의하기`를 눌러 아이디어 ID와 함께 메시지 페이지로 이동한다.
4. 기존 문의가 있으면 해당 thread를 연다.
5. 기존 문의가 없으면 첫 메시지를 통해 신규 thread를 생성한다.
6. 이후 메시지를 주고받고 문의 상태와 unread를 목록에서 확인한다.

### 10.7 프로필 수정

1. 마이페이지에서 프로필 수정 모달을 연다.
2. 기존 이름, 프로필 ID, 이미지가 form에 로드된다.
3. 입력 또는 이미지를 변경하면 미리보기가 즉시 바뀐다.
4. 유효성 검사를 통과한 뒤 저장한다.
5. mock 사용자와 `db.me`가 함께 갱신된다.
6. 헤더와 마이페이지가 새 프로필로 다시 렌더링된다.

## 11. 데이터·상태 관리 방식

### 11.1 서버 상태

- TanStack React Query를 사용한다.
- 주요 query key:
  - 현재 사용자
  - 아이디어 목록
  - 아이디어 상세
  - 보유 Seed Unit
  - 포인트 원장
  - 메시지 thread
- mutation 성공 후 관련 query를 invalidate하거나 사용자 데이터를 직접 cache에 설정한다.

### 11.2 mock 서버

- MSW가 `/api` 요청을 처리한다.
- mock 데이터는 실행 중 메모리에서 변경된다.
- 새로고침 또는 mock worker 재시작 시 일부 변경 사항이 초기 데이터로 돌아갈 수 있다.
- 프로필 이미지는 Data URL이므로 실제 파일 서버나 object storage에는 업로드되지 않는다.
- 실제 운영 서버로 전환할 때는 인증 cookie/session, 파일 업로드, 영속 DB, 권한 검증을 서버에서 구현해야 한다.

### 11.3 권한 표현

- 아이디어별 `capabilities` 값으로 다음 동작의 가능 여부를 제어한다.
  - 편집
  - 피드백
  - 구매
  - 기업 관심
  - 열람 요청
  - 열람 요청 검토
- 클라이언트 UI의 비활성화만으로 보안을 보장하지 않으며 실제 서버에서도 동일한 권한 검증이 필요하다.
